423 #if KMP_HANDLE_SIGNALS
424  /*
425  Signal handling is disabled by default, because it confuses users: In case of sigsegv
426  (or other trouble) in user code signal handler catches the signal, which then "appears" in
427  the monitor thread (when the monitor executes raise() function). Users see signal in the
428  monitor thread and blame OpenMP RTL.
429 
430  Grant said signal handling required on some older OSes (Irix?) supported by KAI, because
431  bad applications hung but not aborted. Currently it is not a problem for Linux* OS, OS X* and
432  Windows* OS.
433 
434  Grant: Found new hangs for EL4, EL5, and a Fedora Core machine. So I'm putting
435  the default back for now to see if that fixes hangs on those machines.
436 
437  2010-04013 Lev: It was a bug in Fortran RTL. Fortran RTL prints a kind of stack backtrace
438  when program is aborting, but the code is not signal-safe. When multiple signals raised at
439  the same time (which occurs in dynamic negative tests because all the worker threads detects
440  the same error), Fortran RTL may hang. The bug finally fixed in Fortran RTL library provided
441  by Steve R., and will be available soon.
442  */
